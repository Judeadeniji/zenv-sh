---
title: Share secrets with your team
description: Set up organizations, invite members, and manage access to shared secrets.
---

## Create an organization

Every team starts with an org. Create one from the CLI:

```bash
zenv orgs create --name "Acme Corp"
```

This returns an org ID you'll use for everything below.

## Invite members

Add team members with a specific role:

```bash
zenv orgs add-member ORG_ID --email teammate@acme.com --role dev
```

Available roles:

| Role | Access |
|------|--------|
| `admin` | Full access — manage members, projects, and secrets |
| `senior_dev` | Create projects, manage secrets across all environments |
| `dev` | Read/write secrets in assigned projects |
| `contractor` | Read-only access to assigned projects |
| `ci_bot` | Automated access — read secrets, no org management |

## Create a shared project

Projects belong to orgs. Create one and link it to your directory:

```bash
zenv projects create --org ORG_ID --name "api-server"
zenv projects init PROJECT_ID --env production
```

## How team encryption works

When you invite a member, zEnv uses X25519 asymmetric encryption to grant them access:

1. The invitee's public key is fetched from the server
2. Your client wraps the project's DEK with their public key
3. The wrapped key is stored server-side
4. When they unlock, their private key unwraps the DEK

The server never sees the plaintext DEK. Each member decrypts independently using their own Vault Key.

## Remove a member

```bash
zenv orgs remove-member ORG_ID MEMBER_ID
```

Their wrapped key is deleted immediately. They lose access on their next unlock attempt — existing cached sessions continue until they expire.

## List members

```bash
zenv orgs members ORG_ID
```

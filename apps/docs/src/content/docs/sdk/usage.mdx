## Basic setup

```typescript
import { zenv } from "@zenv/sdk";

const vault = zenv({
  token: process.env.ZENV_TOKEN!,
  vaultKey: process.env.ZENV_VAULT_KEY!,
  projectId: process.env.ZENV_PROJECT_ID!,
  environment: "production",
});
```

## Loading secrets

### With a schema

`load()` with a schema fetches only the keys defined in the schema, decrypts them locally, and validates each value. This is the recommended approach for application startup.

```typescript
import { z } from "zod";

const vault = zenv({
  token: process.env.ZENV_TOKEN!,
  vaultKey: process.env.ZENV_VAULT_KEY!,
  projectId: process.env.ZENV_PROJECT_ID!,
  environment: "production",
  schema: z.object({
    DATABASE_URL: z.string().url(),
    STRIPE_API_KEY: z.string().min(1),
    PORT: z.string().transform(Number),
  }),
});

const secrets = await vault.load();
// secrets.DATABASE_URL  — string (validated as URL)
// secrets.STRIPE_API_KEY — string (validated as non-empty)
// secrets.PORT — number (transformed by Zod)
```

If any secret is missing or fails validation, `load()` throws with **all** errors at once:

```
[zEnv] Startup validation failed:
  DATABASE_URL — Invalid url
  PORT — Expected number, received string

Application did not start. Fix the above and retry.
```

### Without a schema

`load()` without a schema fetches **all** secrets for the project and environment. Values are returned as raw strings with no validation.

```typescript
const vault = zenv({
  token: process.env.ZENV_TOKEN!,
  vaultKey: process.env.ZENV_VAULT_KEY!,
  projectId: process.env.ZENV_PROJECT_ID!,
  environment: "production",
});

const secrets = await vault.load();
// Record<string, string> — every secret in the project
```

This makes two API calls (list + bulk fetch). If you know which keys you need, prefer a schema or `select()`.

## Selecting specific secrets

When you have many secrets but only need a few, use `select()`. It makes a single bulk request regardless of how many keys you ask for.

Three calling styles:

```typescript
// Variadic strings
const secrets = await vault.select("DATABASE_URL", "REDIS_URL");

// Array
const keys = ["DATABASE_URL", "REDIS_URL"];
const secrets = await vault.select(keys);

// Inline schema — extracts keys, fetches, validates
const secrets = await vault.select(
  z.object({
    DATABASE_URL: z.string().url(),
    REDIS_URL: z.string(),
  }),
);
```

### Validation behavior

- **Constructor schema defined** — `select("KEY1", "KEY2")` automatically validates each value against the matching field in the constructor schema. You don't pass the schema twice.
- **Inline schema** — `select(z.object({...}))` uses the inline schema for validation, ignoring the constructor schema.
- **No schema anywhere** — values are returned as raw strings.
- **`disableValidation: true`** — all validation is skipped regardless.

## Individual operations

### `get(name)`

Fetch and decrypt a single secret. One HTTP request per call — use `select()` if you need multiple.

```typescript
const dbUrl = await vault.get("DATABASE_URL");
```

**Validation:** If the constructor has a schema and validation is enabled, `get()` validates the returned value against the matching schema field. A `DATABASE_URL` with `z.string().url()` will throw if the decrypted value isn't a valid URL.

**Strict mode** (default: `true`): Rejects keys not defined in the schema.

```typescript
await vault.get("DATABASE_URL"); // OK — defined in schema
await vault.get("UNKNOWN_KEY");  // throws: 'UNKNOWN_KEY' is not defined in your schema
```

Set `strict: false` to allow fetching any key.

### `set(name, value)`

Encrypt a value locally and store it. Creates the secret if it doesn't exist, updates it otherwise (auto-increments version).

```typescript
await vault.set("DATABASE_URL", "postgres://user:pass@host:5432/db");
```

### `delete(name)`

Delete a secret and all its versions.

```typescript
await vault.delete("OLD_API_KEY");
```

## Schema validation

The SDK supports any [Standard Schema](https://github.com/standard-schema/standard-schema) compliant library:

```typescript
// Zod
import { z } from "zod";
const schema = z.object({
  DATABASE_URL: z.string().url(),
  API_KEY: z.string().min(1),
});

// Valibot
import * as v from "valibot";
const schema = {
  DATABASE_URL: v.pipe(v.string(), v.url()),
  API_KEY: v.pipe(v.string(), v.minLength(1)),
};

// ArkType
import { type } from "arktype";
const schema = {
  DATABASE_URL: type("string"),
  API_KEY: type("string > 0"),
};
```

### How validation applies across methods

| Method | Schema defined | `disableValidation` | Behavior |
|---|---|---|---|
| `load()` | Yes | No | Fetches schema keys, validates each value |
| `load()` | No | — | Fetches ALL secrets, returns raw strings |
| `select("K1")` | Yes (constructor) | No | Fetches K1, validates against constructor schema |
| `select(schema)` | Any | No | Fetches schema keys, validates against inline schema |
| `select("K1")` | No | — | Fetches K1, returns raw string |
| `get("K1")` | Yes | No | Fetches K1, validates against schema field |
| `get("K1")` | No | — | Fetches K1, returns raw string |
| Any | Any | Yes | All validation skipped, raw strings returned |

### Disabling validation

Use schema as a fetch manifest only — values pass through as raw strings:

```typescript
const vault = zenv({
  schema: z.object({ DATABASE_URL: z.string() }),
  disableValidation: true,
  // ...
});
```

## Configuration reference

```typescript
interface ZEnvConfig {
  /** Service token for API authentication (required). */
  token: string;

  /** Vault Key for local key derivation (required). Never sent to server. */
  vaultKey: string;

  /** Project ID (required). */
  projectId: string;

  /** Environment name. Defaults to "development". */
  environment?: string;

  /** Schema — defines which secrets to fetch and how to validate them. */
  schema?: StandardSchema;

  /** Strict mode (default: true). Rejects get()/select() for keys not in schema. */
  strict?: boolean;

  /** Skip all schema validation. Schema still used as fetch manifest. */
  disableValidation?: boolean;

  /** API base URL. Defaults to "https://api.zenv.sh". */
  baseUrl?: string;
}
```

## Error handling

The SDK throws descriptive errors with context needed to fix the problem.

**Missing credentials:**

```
[zEnv] Missing ZENV_TOKEN. Set it in your environment:
  export ZENV_TOKEN=ze_...
```

**Missing secrets:**

```
[zEnv] Startup validation failed:
  DATABASE_URL — secret not found in 'production'
  STRIPE_KEY — secret not found in 'production'

Application did not start. Fix the above and retry.
```

**Validation failures:**

```
[zEnv] Validation failed for 'DATABASE_URL': Invalid url
```

**Browser detection:**

```
[zEnv] @zenv/sdk detected a browser environment (window is defined).
ZENV_TOKEN and ZENV_VAULT_KEY are server credentials — they must never
reach the browser.
```

All `load()` and `select()` errors are collected and reported at once — the SDK doesn't fail on the first missing secret and hide the rest.

## TypeScript inference

`load()` infers its return type from the schema. Zod's `.transform()` is reflected in the output type:

```typescript
const schema = z.object({
  PORT: z.string().transform(Number),
  DEBUG: z.string().transform((v) => v === "true"),
});

const secrets = await vault.load(schema);
// secrets.PORT  — number
// secrets.DEBUG — boolean
```

Without a Standard Schema validator, all values are typed as `string`.
